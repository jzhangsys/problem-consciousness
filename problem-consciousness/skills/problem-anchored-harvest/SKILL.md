# 問題錨定 harvest:問題當種子,不是窄應用詞

## 核心紀律(本專案的教訓固化)
不同來源的描述層級不同,**種子要配合來源的語言**:
- **補助/基礎研究**:寫機制/問題層(某底層傳輸機制、某增強機制、某界面阻抗),**很少寫應用**(某終端產品場景)。用窄應用詞撈 → 漏掉上游領先指標。
- **專利**:寫結構/方法 + CPC。
- **產業新聞**:寫產品/公司/供應鏈。

## 種子設計(每個驗證過的子問題 → 查詢)
**機制層查詢 + 應用脈絡錨**(兩者都要,寬度控制見 `problem-framing`):
- 每個瓶頸 → 機制層措辭(該領域文獻怎麼描述這個機制)+ 應用脈絡錨(該機制落地的產品/場景詞)。
- 機制詞負責召回上游基礎研究,應用錨負責把範疇收斂回目標場景。
加上**上游賦能查詢**(補助常見):相鄰基礎機制、新興製程、可靠度等橫向詞。

## 語意擴增(非字面)
harvest 回來後 → 嵌入**同一 whitened 流形** → 用語意鄰近**指派/篩選**到技術區域(`embed_lib` + 最近區域質心),自動往目標技術/應用脈絡靠攏。**不靠關鍵詞字面匹配**——避免漏掉「講同一問題但用不同詞」的文件。

## 適用 + 驗真
- 各源(paper/patent/conference/funding/news)皆以此設計種子。
- 來源若被 SSL/bot 擋(如 .gov.tw、Google Patents),改真實瀏覽器(見專案既有瀏覽器 harvest 模式)。
- harvest 後務必過 `manifold-corpus-control`(去重/驗真/年份覆蓋)與相關性閘,再進嵌入。

# 問題意識層(主控)

## 立場(必背)
核心問題是**根輸入**,不是某一格的配角。白地=「核心痛點×學術動能×解法稀疏」、物理 headroom=「某痛點的物理上限」、預測目標=「某痛點未來進場」、harvest 種子=「問題」——全部往上追都是「問題怎麼框」。**框錯一次,下游全錯一次**(例:用窄應用詞當 funding 種子→漏掉上游基礎補助→測不出領先訊號)。

## 何時啟動(觸發)
- 換領域 / 換語料 / 新主題 → 必先框問題。
- 要設計 harvest 種子、要定義白地、要框機會/未來路徑 → 必先跑本層並用其輸出當種子。

## 流程(辨識→拆解→驗證→傳播)
1. **辨識**(從可量化瓶頸,非模糊主題)→ `problem-identification`
2. **拆解**(系統→元件→機制→物理極限,沿系統端到端的因果/阻抗鏈)→ `problem-decomposition`
3. **驗證**(是真瓶頸嗎?物理+Amdahl+部署落差)→ `problem-validation`
4. **錨定傳播**(問題當種子餵 harvest/嵌入/白地)→ `problem-anchored-harvest`

## 寬度控制(核心紀律)
框在「**可量化瓶頸 × 應用脈絡**」這個中間層:
- ❌ 太窄(「某個窄應用場景的整體效率」)→ 漏上游機制/基礎研究。
- ❌ 太廣(「整個大領域」)→ 抓進無關的次領域。
- ✅ 正確(「某界面層瓶頸指標 @ 高負載工況,鎖定特定目標應用」)。

## 執行層
由支援腳本實作:LLM bottom-up 挖痛點 + 逐痛點剖面 + 每痛點物理上限。本 skill 治理它們,確保問題框得對、夠深、是真瓶頸、且正確傳播。

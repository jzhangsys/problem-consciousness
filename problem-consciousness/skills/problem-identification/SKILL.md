# 問題辨識:模糊主題 → 可量化瓶頸

## 規則:一個合格的「核心問題」必須四欄齊全
| 欄位 | 例(合格) | 反例(駁回) |
|---|---|---|
| 指標+單位 | 領域原生指標 (原生單位) | 「效能」(無指標) |
| 當前部署值 | 12–30 | — |
| 物理/理論目標 | 理論極限 ~3 | — |
| 尺度 | 元件/局部級 (領域相關尺度) | 「整體效能」(尺度不明) |

缺指標或缺尺度 → **不是核心問題,是主題**,打回。

## 兩條萃取來源(交叉)
1. **bottom-up 語料**:逐篇問題拆解 → 痛點聚類(`core_problems.py`/`painpoint_profile.py`)。給「業界正在喊什麼痛」。
2. **top-down 物理**:沿系統端到端的瓶頸鏈,每段問「這段的量化極限是什麼」(`physics_kernel`)。給「物理上哪裡還有空間」。

兩者交集 = **真實且有物理空間的核心問題**;只有 bottom-up = 可能是喊爽的;只有 top-down = 可能沒人在乎。

## 輸出
核心問題清單,每條附四欄 + 來源(語料/物理/兩者)。傳給 `problem-decomposition`。
示例骨幹痛點:沿領域核心瓶頸鏈拆出的一組可量化痛點(每條 = 指標 + 單位 + 尺度),數量視領域而定。

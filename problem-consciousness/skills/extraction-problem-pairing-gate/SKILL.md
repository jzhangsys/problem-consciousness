# [抽取·入選判準] 涵蓋問題=痛點+解法+對應 齊備(非命中某節點)

## 由來(2026-06-18 用戶定調)
某個鄰近工程子領域(器件/模組層)「算不算目標問題範疇」?判準不是領域標籤,也不是「命中某節點」,而是:**該件有沒有清楚表述問題意識的三件——**
1. **痛點(實體問題)**:具體、可定位的問題(如接點疲勞的循環壽命 cycles-to-failure、某路徑冗餘不足、瞬態負載致峰值失效變數)。
2. **解法(概念問題)**:對應的機制/做法(如低溫接合製程、雙面/對稱式強化、緩衝層吸收瞬態)。
3. **對應清晰**:痛點 ↔ 解法 的組合被清楚對上(這個解法是在解這個痛點),不是各說各話。

三者齊備 → **在範疇**(即使是功率電子);缺對應 / 只有一邊 / 含糊 → **排除**。

## 兩條正交軸(別混淆)
- **pc_complete(入選軸)**:痛點+解法+對應齊備?→ 決定「在不在範疇、要不要保留」。
- **anchor_correct(錨定軸)**:這件的痛點↔解法是否真打到它被錨的那個 problem_id 的瓶頸?→ 決定「錨對了沒」。
- **四種組合**:
  - 齊備 + 錨對 → 入選,正常折入。
  - **齊備 + 錨錯 → 入選但 re-anchor**(移到正確節點;這正是大量「relevance=no」的真相——有效但錯錨,不可當離題丟)。
  - 不齊 + 任何 → 排除(標 incomplete,不折入 dossier 材料)。

## 實作要點
- 抽取輸出除 relevance(=anchor_correct,是否命中該節點瓶頸)外,須判 **pc_complete**(痛點✓解法✓對應✓)與 **suggested_node**(若錨錯,建議正確節點)。
- 「對應清晰」是語意判斷,須 LLM 裁定(非只看兩欄非空);兩欄非空只是必要非充分。
- relevance=no 的件:先過本閘——pc_complete 者 → re-anchor 候選(交 [[anchor 機制]]/`_anchor_flagger`/id_overrides);不齊者才排除。
- 馬拉松全量處理時:**入選用 pc_complete,不用 node-hit**;錯錨的回流 re-anchor,別灌錯節點也別誤丟有效件。
